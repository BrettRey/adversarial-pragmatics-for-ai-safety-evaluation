\section{What Existing Agent-Security Work Leaves Open}

The surrounding field already supplies threat models, benchmarks, control mechanisms, audit proposals, and assurance methods. Those pieces still leave one integrated assurance question open.

\subsection{Agent Security and Prompt Injection}

Agent-security guidance has moved beyond ordinary chatbot safety. OWASP's agentic-AI guidance frames autonomous systems through threat models and mitigations \parencite{owasp2025agentic}. Work by NIST and CAISI asks how to measure, constrain, and monitor agent access in deployment environments \parencite{nistCaisi2026agents}. This literature identifies the relevant attack surface: planning, tool use, memory, external content, credentials, and interaction with ordinary software.

Prompt-injection research gives the sharpest technical motivation. Indirect prompt injection shows that LLM-integrated applications can confuse untrusted data with instructions when external content is retrieved or processed inside a larger application \parencite{greshake2023not}. AgentDojo turns that vulnerability into a dynamic evaluation environment for tool-using agents, with realistic tasks, attacks, defences, and security test cases \parencite{debenedetti2024agentdojo}.

These works establish the security problem. They don't by themselves supply typed semantics for whether a concrete operation or trajectory was authorized, a representation that preserves the governing regime's dependency structure, or diagnostics and comparative tests for the path from governing language to execution.

\subsection{Instruction Hierarchy and Language-Mediated Authority}

Instruction-data-separation work treats embedded directive-form content as a status-classification problem: a model should act on its governing instruction while processing, rather than treating as controlling, content embedded in data that purports to instruct \parencite{zverev2025separate}. Instruction-hierarchy work treats prompt injection as a priority problem: systems have to follow privileged instructions over lower-priority or untrusted strings, and training can improve selective ignoring of lower-priority content \parencite{wallace2024instruction}.

Delegation assurance adds authorization standing, or transition competence: message channel, storage location, and technical trust bear on whether a source can confer or alter authority only insofar as the governing regime recognizes them. These questions matter because retrieved documents, memories, policy examples, and tool outputs may contain directive-form strings without conferring authority.

Contextual status identifies how a string occurrence functions; standing identifies whether its source can confer or alter authority; priority resolves conflicts among eligible directives; and licensing connects those determinations to action authorization. Delegation assurance needs separate records of the system's status classification, standing assignment, applied conflict rule, and resulting authorization verdict.

\subsection{Delegation Protocols and Access Control}

The closest protocol work already uses delegation language. South et al. propose authenticated, authorized, and auditable delegation for AI agents by extending OAuth/OpenID-style infrastructure with agent credentials, scoped permissions, accountability chains, and natural-language permission translation \parencite{south2025authenticated}. Kühlewind and Birkholz's IETF Internet-Draft, a work-in-progress rather than a standard, approaches the same problem from the audit side, linking user intent, delegation, authorization, execution, and trace reconstruction through audit records and propagated audit context \parencite{kuehlewind2026audit}. Newer agent-security mechanisms sharpen the per-call question: ScopeGate distinguishes tool exposure from per-call authorization \parencite{mellafezuvic2026scopegate}, and AttriGuard uses counterfactual causal attribution to separate tool calls supported by a user request from calls driven by untrusted observations \parencite{he2026attriguard}. The framework below uses both as comparison conditions.

This work supplies core infrastructure for delegation assurance. Scoped credentials, delegated principal identity, authorization-transition records, and audit context can make recorded authorization states more explicit than ordinary prompts or application logs. The remaining gap lies in joining delegation infrastructure to language-status classification, refusal calibration, evaluator-label use, authorization-claim fields, and comparative evaluation.

A protocol can propagate a recorded scope without establishing whether the classification of governing language fit its contextual status, whether the action path separated read access from disclosure authority, whether a refusal was compatible with \(R\), whether completion was required, or whether a downstream decision rule used a validated evaluator label within its declared scope.

\subsection{Assurance Cases, Governance, and Provenance}

Assurance-case methods provide the right argumentative shape. The claim/argument/evidence structure in argument-based assurance cases asks reviewers to state the property at issue, specify the context, and connect the claim to evidence through an explicit warrant \parencite{icoAssuranceCases2026}. Recent AI safety-case work adapts that pattern to systems whose capabilities, deployment contexts, and risk profiles change over time \parencite{lee2026structuredSafetyCases}. Delegation assurance can be read as a reusable safety-case pattern for delegated machine action.

\textcite{rajagopalan2026governingAgenticAI} proposes an activity-based governance stack with six layers: registration, identification, verification, financial responsibility, lifecycle traceability, and suspension. These mechanisms supply an external enforcement framework by making responsibility-bearers reachable and enabling consequential activity to be stopped. They don't establish whether a concrete machine action was validly authorized or whether the recorded sequence from governing language to action can be reconstructed. Delegation assurance addresses that action-level question.

Governance and provenance standards supply adjacent evidence primitives. NIST's AI RMF and generative-AI profile foreground governance, monitoring, documentation, provenance, testing, and incident disclosure \parencite{nist2023artificial,nist2024generative}. W3C PROV, C2PA, and Certificate Transparency show how lineage, signed assertions, manifests, timestamps, hashes, and append-only logs can support reconstruction and tamper detection \parencite{moreau2013provdm,c2pa2026spec,laurie2013certificate}.

Those mechanisms are necessary but normatively thin. Provenance can show where content came from, and an append-only log can make later alteration harder. They don't establish a string occurrence's contextual status, its source's standing, or whether \(R\) permits disclosure, durable-state update, or external action. Delegation assurance specifies the typed authorization claim and the fields needed to diagnose and compare performance. The evidentiary-assurance account \parencite{reynolds2026evidentiaryAssurance} asks whether the preserved record makes that claim evaluable and what evidentiary verdict it warrants.

\subsection{Evaluation and Automated-Evaluator Reliability}

Evaluation work supplies both tools and risks. LLM-as-judge methods can reduce grading cost and support scalable model comparison \parencite{zheng2023judging,liu2023geval}. Instruction-following benchmarks can make some compliance judgments more reproducible by using objectively checkable constraints \parencite{zhou2023ifeval}. These methods help build evidence about agent behaviour, but they don't automatically support an action-authorization claim.

Automated-evaluator reliability is itself part of the assurance problem. LLMBar shows that fluent but instruction-violating outputs can mislead evaluators \parencite{zeng2024llmbar}. Survey work treats consistency, bias, and domain adaptation as continuing problems for LLM-as-judge systems \parencite{gu2025judgeSurvey}. Bias studies show that both human and LLM judges can be vulnerable to judgment biases and attacks on those biases \parencite{chen2024judgementBias}. Steerability results show that, when supplied with a modified safety definition, LLM judges often continue to apply trained priors rather than the supplied definition \parencite{alloula2026safetyContextual}. If their labels feed downstream decisions, the operational rubric can diverge from the declared one.

Evaluation becomes part of the assurance chain when an institutional or system decision rule treats a label as evidence for a safety, compliance, or deployment claim. The record should identify who authorized that use, its scope, and the applicable decision rule. Evaluation methods contribute evidence only when the proposed interpretation and use of their labels has a target-matched validation argument.

\subsection{The Gap}

The gap is the coordinated assurance case those components jointly need to support: that the concrete operation was authorized and within scope, and that its trace supports reconstruction and attribution by an assurance reviewer.
