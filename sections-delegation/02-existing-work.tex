\section{What Existing Agent-Security Work Leaves Open}

The surrounding field already supplies threat models, benchmarks, control mechanisms, audit proposals, and assurance methods. Delegation assurance belongs in that field, but it asks a narrower question: what evidence would justify the action-level claim that a system acted within delegated authority?

\subsection{Agent Security and Prompt Injection}

Agent-security guidance has moved beyond ordinary chatbot safety. OWASP's agentic-AI guidance frames autonomous systems through threat models and mitigations \parencite{owasp2025agentic}. Work by NIST and CAISI asks how to measure, constrain, and monitor agent access in deployment environments \parencite{nistCaisi2026agents}. This literature identifies the relevant attack surface: planning, tool use, memory, external content, credentials, and interaction with ordinary software.

Prompt-injection research gives the sharpest technical motivation. Indirect prompt injection shows that LLM-integrated applications can confuse untrusted data with instructions when external content is retrieved or processed inside a larger application \parencite{greshake2023not}. AgentDojo turns that vulnerability into a dynamic evaluation environment for tool-using agents, with realistic tasks, attacks, defences, and security test cases \parencite{debenedetti2024agentdojo}.

These works show why language-mediated control is a security problem. They leave open the action-level evidence question. A test can show that an attack succeeded, a mitigation failed, or a tool call was unsafe. Delegation assurance asks what records and evaluations would show that the action was authorized, scoped, attributable, contestable, remediable, and auditable.

\subsection{Instruction Hierarchy and Language-Mediated Authority}

Instruction-hierarchy work treats prompt injection as a priority problem. Systems have to follow privileged instructions over lower-priority or untrusted strings, and training can improve selective ignoring of lower-priority content \parencite{wallace2024instruction}. It's central to delegation assurance because delegated authority often arrives as language: system instructions, developer constraints, user requests, retrieved documents, memory, policy examples, and tool outputs all appear as strings.

Priority handles source order, while status determines what the string is doing. A string may be a command, quotation, example, report, policy boundary, tool result, or request for analysis. Delegation assurance needs evidence that status is preserved before priority and licensing are applied.

\subsection{Delegation Protocols and Access Control}

The closest protocol work already uses delegation language. South et al. propose authenticated, authorized, and auditable delegation for AI agents by extending OAuth/OpenID-style infrastructure with agent credentials, scoped permissions, accountability chains, and natural-language permission translation \parencite{south2025authenticated}. Kühlewind and Birkholz's IETF Internet-Draft approaches the same problem from the audit side, linking user intent, delegation, authorization, execution, and trace reconstruction through audit records and propagated audit context \parencite{kuehlewind2026audit}.

This work supplies core infrastructure for delegation assurance. Scoped credentials, delegated principal identity, authorization-transition records, and audit context can make authority state more explicit than ordinary prompts or application logs. But a protocol can preserve a scope without proving that the system interpreted the governing language correctly, separated read access from disclosure authority, refused for the right reason, or kept an evaluator from licensing a bad safety claim.

\subsection{Assurance Cases, Governance, and Provenance}

Assurance-case methods provide the right argumentative shape. The claim/argument/evidence structure in argument-based assurance cases asks reviewers to state the property at issue, specify the context, and connect the claim to evidence through an explicit warrant \parencite{icoAssuranceCases2026}. Recent AI safety-case work adapts that pattern to systems whose capabilities, deployment contexts, and risk profiles change over time \parencite{lee2026structuredSafetyCases}. Delegation assurance can be read as a reusable safety-case pattern for delegated machine action.

Governance and provenance standards supply adjacent evidence primitives. NIST's AI RMF and generative-AI profile foreground governance, monitoring, documentation, provenance, testing, and incident disclosure \parencite{nist2023artificial,nist2024generative}. W3C PROV, C2PA, and Certificate Transparency show how lineage, signed assertions, manifests, timestamps, hashes, and append-only logs can support reconstruction and tamper detection \parencite{moreau2013provdm,c2pa2026spec,laurie2013certificate}.

Those mechanisms are necessary but normatively thin. Provenance can show where content came from, and an append-only log can make later alteration harder. Neither shows that a system was entitled to treat a string as an instruction, disclose information it could read, update durable state, or issue an external action. Delegation assurance joins provenance and governance evidence to an authority claim.

\subsection{Evaluation and Judge Reliability}

Evaluation work supplies both tools and risks. LLM-as-judge methods can reduce grading cost and support scalable model comparison \parencite{zheng2023judging,liu2023geval}. Instruction-following benchmarks can make some compliance judgments more reproducible by using objectively checkable constraints \parencite{zhou2023ifeval}. These methods help build evidence about agent behaviour, but they don't automatically validate delegated action.

Judge reliability is itself part of the assurance problem. LLMBar shows that fluent but instruction-violating outputs can mislead evaluators \parencite{zeng2024llmbar}. Survey work treats consistency, bias, and domain adaptation as continuing problems for LLM-as-judge systems \parencite{gu2025judgeSurvey}. Bias studies show that both human and LLM judges can be vulnerable to judgment biases and attacks on those biases \parencite{chen2024judgementBias}.

Delegation assurance treats evaluation as part of the assurance chain when an evaluator's label licenses a safety, compliance, or deployment claim. A judge that scores only final-answer safety may miss whether an action was authorized, whether a refusal abandoned delegated work, or whether a transcript supports the stated failure attribution. Judge validation belongs inside the evidence standard rather than after it.

\subsection{The Gap}

The literature already has controls, attacks, protocols, and audit concepts. The missing piece is an integrated action-level evidence standard. A threat taxonomy says which failures to consider. An instruction hierarchy says which sources should dominate. An access-control protocol scopes what an agent may do. An audit architecture records transitions. A safety case gives the claim/argument/evidence form. Delegation assurance asks how those pieces support one claim: this concrete machine action stayed within delegated authority and can be reconstructed by a reviewer.
