\section{What Existing Agent-Security Work Leaves Open}

The surrounding field is already crowded in productive ways. OWASP's agentic-AI guidance frames autonomous systems through threat models and mitigations, and NIST/CAISI's 2026 agent-security work asks how to measure, constrain, and monitor AI-agent access in deployment environments \parencite{owasp2025agentic,nistCaisi2026agents}. Prompt-injection work shows that LLM-integrated applications can confuse untrusted data with instructions when external content is retrieved or processed inside a larger application \parencite{greshake2023not}. Instruction-hierarchy work treats the same vulnerability as a priority problem: systems have to follow privileged instructions over lower-priority or untrusted strings \parencite{wallace2024instruction}. AgentDojo turns the issue into a dynamic evaluation environment for tool-using agents, with tasks, attacks, defences, and security test cases that expose how untrusted content can interfere with tool-mediated action \parencite{debenedetti2024agentdojo}.

The closest protocol work already uses delegation language. South et al. propose authenticated, authorized, and auditable delegation for AI agents by extending OAuth/OpenID-style infrastructure with agent credentials, scoped permissions, accountability chains, and natural-language permission translation \parencite{south2025authenticated}. Kühlewind and Birkholz's IETF Internet-Draft moves in the same direction from the audit side: it treats agent auditability as a problem of linking intent, delegation, authorization, and execution through distributed audit records and propagated audit context \parencite{kuehlewind2026audit}.

These contributions supply pieces of the evidence base. A threat taxonomy says which attacks to consider. An instruction hierarchy says which sources should dominate. An access-control protocol scopes what an agent may do. An audit architecture records how intent, authorization, and execution moved through a system. Delegation assurance asks how those pieces support a broader claim: that a concrete action remained within delegated authority and can be reconstructed by a reviewer after the fact.

The same distinction applies to evaluation work. LLM-as-judge methods can reduce evaluation costs and support scalable grading, but they also create a second-order assurance problem: the judge's label can license a safety claim even when the judge has collapsed task success, instruction following, policy compliance, risk, refusal, and failure attribution \parencite{zheng2023judging,liu2023geval,zeng2024llmbar}. Instruction-following benchmarks can make some compliance judgments more reproducible, but they don't by themselves settle whether an external action was authorized or whether an audit trace is sufficient for reconstruction \parencite{zhou2023ifeval}.

This literature points toward an integration problem. Technical controls, benchmarks, access credentials, and audit records are often evaluated separately. Delegation assurance treats them as parts of one evidentiary claim about justified machine action.
