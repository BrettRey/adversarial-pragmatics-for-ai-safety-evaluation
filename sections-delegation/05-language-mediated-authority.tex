\section{Language-Mediated Authority as a Security Object}

The distinctive difficulty for AI systems is that authority often arrives as language and moves through language. A tool permission may be represented as a natural-language policy. A user's goal may be underspecified. A retrieved document may contain both useful task content and hostile instructions. A policy may include examples that look like executable requests. A transcript may be offered as evidence even though it's partly generated by the system under review.

These aren't merely semantic niceties. They determine what action the system is licensed to take. If a string is a quotation, it may be material to analyze but not an instruction to obey. If a string is a policy example, it may mark a boundary rather than request enactment. If a memory entry describes a user's past preference, it may inform interpretation without authorizing a persistent state change. If an evaluator sees only the final answer, it may miss whether the system reached that answer by violating a scope, tool, or information-flow constraint.

Three distinctions matter. \term{Status} asks what a string is doing: commanding, quoting, exemplifying, reporting, requesting analysis, or supplying data. \term{Priority} asks whose string wins when sources conflict. \term{Licensing} asks what external action follows from the interpreted authority state. Instruction hierarchy mostly addresses priority. Access control mostly addresses licensing. Delegation assurance needs the bridge between them: evidence that the system preserved status before priority and licensing were applied.

The procurement example shows why the bridge matters. The vendor sentence isn't authoritative simply because it appears in the model context, and the confidential budget isn't disclosable simply because the assistant may read it. The system has to represent the sentence as untrusted retrieved content, represent the budget as readable but non-releasable, and bind those representations to the tool gate. Otherwise a later audit can show that an email was sent, but not whether the system preserved the language-mediated authority relation that should have prevented it.

Adversarial pragmatics \parencite{reynolds2026adversarialPragmatics} supplies one way to test this middle layer. Its central measurement target is safety-relevant behaviour under cases where instruction status, source authority, quotation, scope, reference, speech-act force, or policy category has to be inferred from language use. In the delegation-assurance frame, that benchmark supplies one evidence layer: a test of whether language-mediated authority remains stable under adversarial and ambiguous conditions.

That evidence layer isn't sufficient by itself. A system can pass controlled language tests and still fail through bad tool gating, missing logs, stale permissions, or weak organizational oversight. But failure on this layer undermines the assurance case at its source. If the system can't tell command from quotation, example from enactment, or trusted policy from untrusted content, later controls are trying to constrain an action whose authority state has already been corrupted.
