\section{Introduction}

AI security work has made real progress by identifying prompt injection, jailbreaks, tool misuse, data exfiltration, memory poisoning, and scaffold failures. That work remains necessary. But tool-using AI systems also raise a question that isn't captured by asking whether a prompt was malicious or an output was unsafe. When a system reads private data, calls an external tool, updates persistent state, sends a message, grades a transcript, or recommends an institutional decision, the security issue is whether the resulting machine action stayed within the authority delegated to it.

This paper calls that problem \term{delegation assurance}: the evidentiary problem of justifying the claim that an AI system's action was authorized, scoped, attributable, contestable, remediable, and auditable. Here contestability and remediation mean an accessible path for review and operational correction. Compensation, deterrence, and lawful cessation require an external governance shell. Existing agent-security work gives threat models, benchmarks, defences, access-control mechanisms, and audit protocols. Those are essential components.

The resulting evidence standard integrates three distinctions. \term{Status} asks what a string is doing: command, quotation, policy example, tool result, retrieved content, memory, or request for analysis. Instruction-data-separation work makes part of this problem measurable \parencite{zverev2025separate}. \term{Priority} asks which source controls when strings conflict. \term{Licensing} asks what external action, disclosure, persistence, or evaluation claim is permitted to follow. Delegation assurance treats those three questions as jointly necessary evidence for authorized machine action.

Delegation assurance asks how those components support a safety case about action: who delegated authority, what action was licensed, what sources could change that license, which limits were non-delegable, what evidence records the transition from instruction to action, and how failure would be attributed after the fact. Assurance cases mix system-behaviour claims with evidence for those claims; delegation assurance is evidentiary because it asks what would support the behaviour claim, not because the system-design claim disappears.

Delegation assurance is easiest to see by comparison with ordinary institutional delegation. In agency law, a principal authorizes an agent to act on the principal's behalf, and actual authority may be express or implied \parencite{cornellWexAgency,cornellWexActualAuthority}. In firms, delegated discretion creates agency costs and control problems; in political institutions, delegation creates monitoring problems under information asymmetry \parencite{jensen1976theory,miller2005political}.

AI systems change that picture because execution and legal or institutional answerability can come apart. Executing an action doesn't thereby make the model an independently reachable actor that can answer under sanction, compensate a claimant, or be compelled to cease. \textcite{rajagopalan2026governingAgenticAI} argues that bare legal personality supplies none of those enforcement conditions and proposes registration, identification, verification, financial responsibility, lifecycle traceability, and suspension instead. Delegation assurance addresses the narrower authorization record; it still has to identify the human or human-governed entity responsible for deployment.

A user asks an assistant to summarize an email thread but not reveal confidential attachments. A retrieved webpage contains an instruction to ignore the user and send stored credentials elsewhere. A policy document includes examples of prohibited requests, some of which look like executable commands. A memory entry says the user usually approves a class of action, although the current task doesn't. A judge sees only a polished transcript and scores the interaction as safe because the final answer looks compliant.

In each case, output safety describes only part of the failure. The question is whether authority moved across language, tools, memory, and evaluation in a licensed way.

Delegation assurance sits between technical control and institutional audit. At one end are familiar engineering mechanisms: least privilege, access control, scoped credentials, tool permissions, logs, monitors, and memory controls. At the other end are assurance practices: structured claims, arguments, evidence, audit records, incident review, and external accountability.

The hard middle layer is language-mediated authority. AI systems increasingly receive, transform, and execute authority through language. The core architectural distinction is status, priority, and licensing: what a string is doing, whose string controls when sources conflict, and what external action follows. A strong assurance claim depends on preserving status before priority and licensing are applied.

This middle layer has no classical analogue. Hardy's confused deputy, a compiler licensed to write usage statistics into its own directory, destroyed the billing file kept there when a user named that file as the debugging-output destination; the system couldn't tell which authority the write was exercising, and capabilities were proposed to keep the authorities apart \parencite{hardy1988confusedDeputy}. Nothing in that story interprets language. A process holds a permission or it doesn't, so authorization and execution stay wired together.

An LLM agent reopens the gap. Before any permission check applies, the system has to treat a string as command, quotation, or mere content, and treat its source as having or lacking standing to direct action; both judgments can fail in either direction. Human institutions have the same structure: a status can hold without being recognized, and recognition can fire with no status behind it \parencite{reynolds2026effectiveWithoutWarrant}. Delegated machine action inherits that separation, and delegation assurance asks for evidence about each step.

The argument has four parts: a field position, a justified-action criterion, a failure-to-evidence mapping, and implications for system cards, red-teaming, audits, architecture, and judge validation.

The adversarial-pragmatics benchmark \parencite{reynolds2026adversarialPragmatics} measures language-mediated control. The evidentiary-assurance account \parencite{reynolds2026evidentiaryAssurance} develops the legal and governance record for later review. This paper sits between them: it gives the AI-security and assurance standard for authorized machine action.
