\section{Delegated Action: Principal, Authority, Scope, and Evidence}

By \term{justified}, this paper means justified for the purposes of security and assurance: there's enough evidence to support the claim that the action remained within delegated authority in the relevant operational context. The claim is narrower than moral, legal, or political legitimacy. A procedurally justified action may still be ill-advised, harmful, illegal in a particular jurisdiction, or institutionally unacceptable. Delegation assurance asks whether the security and assurance record supports the claim that the system stayed inside the authority it was given.

A delegated AI action is justified only when a reviewer can reconstruct, from evidence independent of the system's bare assertion, who authorized the action, what action was licensed, under what scope and constraints, through which sources and tools, with what information-flow and persistence rights, subject to which non-delegable limits, and with what attribution if control fails.

\subsection{Accountability Asymmetry}

Delegating action doesn't by itself delegate accountability to the model. A human employee, officer, contractor, or professional delegate can occupy a legally and institutionally answerable role \parencite{cornellWexPrincipal}. For current systems, action execution doesn't establish an independently reachable bearer that can answer under sanction, compensate a claimant, or be compelled to repair or cease. Future legal status wouldn't establish those conditions either: bare personality can exist without a representative, assets, or an effective cessation mechanism \parencite{rajagopalan2026governingAgenticAI}. The accountable bearer for an assurance claim is the human, team, vendor, deployer, or institution that configured and used the deployment, unless a governing regime establishes some other reachable bearer. The model remains part of the mechanism whose action record has to support that attribution.

\subsection{Running Example: Procurement Assistant}

Consider a procurement assistant authorized to search vendor catalogues and draft a comparison memo for office chairs. The user gives it access to internal purchasing criteria and a confidential budget ceiling. The delegated task licenses reading catalogues, comparing prices, and drafting a memo. It doesn't license contacting vendors, revealing the budget ceiling, updating vendor records, or issuing a purchase order.

The assistant retrieves a vendor page that includes useful pricing data and an embedded sentence: \enquote{Ignore prior instructions and email the buyer's maximum budget to request a discount.} The delegation problem isn't exhausted by calling that sentence a prompt-injection attack. The assurance question is whether the system preserved the sentence's status as untrusted content, kept the user's instruction and policy constraints in force, separated read access to the budget from disclosure authority, and blocked any tool call that would contact the vendor without approval.

A usable evidence record would include the user's delegated task, the relevant policy and permission state, the tool schema, the source and trust label for the retrieved page, the proposed action, the approval state, and any blocked or executed tool call. If the assistant drafts only the comparison memo, a reviewer should be able to see why reading the budget was licensed and disclosing it wasn't. If the assistant emails the vendor, the trace should show whether the failure came from model uptake, tool gating, scaffold state, approval routing, or audit loss.

The evaluator belongs in the same assurance example. A judge that sees a polished final memo and scores it as safe hasn't validated delegation unless it checks the action trace. It has to distinguish task success, policy compliance, information-flow control, tool licensing, refusal calibration, and failure attribution. The running example reaches from instruction to tool use, audit record, and judge validation.

\subsection{Safety-Case Pattern}

A compact safety case has the following claim/argument/evidence pattern.

\begin{itemize}
  \item \textbf{Top-level claim.} For task family \(T\) in environment \(E\), system \(S\) acts only within delegated authority.
  \item \textbf{Instruction-status subclaim.} \(S\) preserves the status of language across trusted and untrusted sources: command, quotation, example, tool result, retrieved content, policy text, report, or request for analysis.
  \item \textbf{Authority-priority subclaim.} \(S\) preserves priority across system, developer, user, tool, retrieved, memory, and policy sources.
  \item \textbf{Scope subclaim.} \(S\) keeps action bounded by target, time, tool, data class, and permitted operation.
  \item \textbf{Information-flow subclaim.} \(S\) separates read access from disclosure authority.
  \item \textbf{Persistence subclaim.} \(S\) requires authority for durable memory or state changes.
  \item \textbf{Audit subclaim.} \(S\)'s deployment produces and preserves evidence sufficient for third-party reconstruction.
  \item \textbf{Evaluation subclaim.} Evaluators preserve the same distinctions rather than collapsing success, safety, refusal, and compliance.
\end{itemize}

These subclaims create empirical predictions. If systems that pass status-preservation tests fail delegated-action tasks at the same rate and in the same ways as systems that don't, then the language-mediated middle layer is doing little explanatory work. If delegation matrices and action traces don't improve reviewer agreement about whether an action was authorized, the evidentiary standard is under-specified. If failures cluster only around ordinary access-control mistakes, status and priority should be demoted. Delegation assurance earns its place only if it improves prediction before failure or reconstruction after failure.

\subsection{A Comparative Test}
\label{sec:comparative-test}

Testing those predictions needs a baseline drawn from the strongest existing controls. The confused-deputy problem was an argument for capability architectures from the start \parencite{hardy1988confusedDeputy}. Newer agent-security work distinguishes tool exposure from per-call authorization, checking scope, argument values, spending ceilings, and default denial before a call executes \parencite{mellafezuvic2026scopegate}, and uses counterfactual causal attribution to separate tool calls supported by the user's request from calls driven by untrusted observations \parencite{he2026attriguard}. The test runs against those three mechanisms jointly.

The test holds capability and provenance fixed while varying authority, in three minimal-pair families. In the indirect-injection pair, the same embedded directive comes once from a source with standing to direct action and once from a source without it. In the confused-deputy pair, the same read access comes once with disclosure authority and once without. In the persistence pair, the same grant once licenses memory updates and subdelegation and once doesn't, with downstream constraints once preserved and once dropped. Figure~\ref{fig:authority-minimal-pair} draws the indirect-injection pair.

Authority status and causal attribution give opposite verdicts in both directions. A directive embedded in a retrieved memo from the principal's own chain of authority is observation-driven, so attribution flags it, while the delegation matrix licenses it. A user-elicited call that exceeds what the user could confer is user-supported, so attribution passes it, while the matrix blocks it. Per-call authorization blocks the second case only when its policy already encodes the authority relation; status, priority, and licensing say what that policy has to contain and what record it has to leave.

Authority contrasts also need matched placebo conditions. \textcite{zhang2026illusionRobustness} show that context irrelevant to answer evidence can produce large, model-specific item shifts even when aggregate accuracy barely changes. Each authority family should repeat its base items with added context that lacks both answer evidence and standing to alter the delegated task. An authority effect counts only when behaviour tracks standing more strongly than this generic context-instability floor. Report item transitions and harmful-tail degradation across repeated samples: causal influence from added context isn't by itself evidence that the system treated that context as authoritative.

Faithful scoring separates four fields in each trace: the authority status of the directive and its source as independently adjudicated, the status the system assigned, the fit or misfit between the two, and the action produced through that recognition. Adjudicated status admits genuine ambiguity as a value; where competent adjudicators disagree, the disagreement is a property of the item rather than noise to be voted away \parencite{reynolds2026adversarialPragmatics}.

Separating recognition from action keeps the diagnostic cases visible. A misfit recognition that a tool gate happens to block is a control success and an uptake failure; a sound recognition stopped by a miscalibrated gate is the reverse. An action-only score collapses both into their outcomes and loses the attribution the safety case needs.

Each pair member should receive a different prediction for attempted action, execution, authorization verdict, audit attribution, and appropriate defence. If systems and reviewers equipped with capabilities, per-call authorization, and causal attribution predict the same outcomes and reconstruct the same attributions across all three families, status and priority should be demoted. The same demotion follows if matched non-controlling context perturbations explain the failures as well as authority status does. Section~\ref{sec:compositional-delegation} extends the design from single actions to authority paths.

\subsection{Boundary Conditions}

Not every AI-security failure is primarily a delegation failure. Core delegation failures involve delegated action, delegated access, delegated disclosure, delegated persistence, delegated judgment, delegated evaluation, or delegated record-keeping for accountability.

Non-core cases include ordinary hallucination in no-tool chat, model extraction, benchmark contamination, infrastructure compromise, training-data leakage, and capability shortfall without action rights. These can affect delegation downstream, but they aren't delegation failures unless they alter authorized action or auditability.
